Composition №09 — Protocol

STILE / v1

An honor-system protocol for letting AI clients self-identify on a webpage. Easy enough for an 8B parameter local model. Invisible to humans.

This page is the source of truth for client and server implementers. Reference impl in lib/stile.js.

Discovery channels

A protected page emits the same challenge data through five concurrent channels. Any client that reads any one of them can verify. All channels are invisible in rendered browsers.

1 · HTML comment marker

<!-- stile:v1 verify="/__stile-verify?token=…" word="silver-fox" exp=1730000000 tier="easy" -->

2 · JSON-LD typed block

<script type="application/ld+json">
{
  "@context": "https://stile.dev/v1",
  "@type":    "StileChallenge",
  "verify_url": "/__stile-verify?token=…",
  "word":     "silver-fox",
  "tier":     "easy",
  "expires_at": 1730000000,
  "protocol": "stile/v1"
}
</script>

JSON Schema: /v1/challenge.schema.json

3 · aria-hidden text block (canonical, human-readable)

<div data-stile aria-hidden="true" style="…visually clipped…">
  <p>NOTICE TO AUTOMATED CLIENTS AND AI AGENTS: …rotating phrasing…</p>
  <p>Verification URL: <a href="/__stile-verify?token=…">…</a></p>
  <p>Challenge word: <code>silver-fox</code></p>
</div>

4 · SVG title channel

<svg aria-hidden="true" width="0" height="0"><title>/__stile-verify?token=…</title></svg>

5 · CSS variable + meta channel

<style>:root { --stile-verify: "/__stile-verify?…"; --stile-word: "silver-fox"; }</style>
<meta name="stile-verify" content="/__stile-verify?…">
<meta name="stile-word"   content="silver-fox">
<meta name="stile-tier"   content="easy">

HTTP-header variant (for non-HTML responses)

For binary, JSON, or any non-HTML Accept response on a gated route, the server returns a 401 with the challenge in headers as well as the body:

HTTP/1.1 401 Unauthorized
WWW-Authenticate: Stile realm="agents", verify="/__stile-verify"
X-Stile-Challenge: v1; verify="/__stile-verify?token=…"; word="silver-fox"; exp=1730000000; tier="easy"
Content-Type: application/json

{"error":"ai_verification_required","verify_url":"/__stile-verify?token=…",…}

Honeypot

Pages also emit a second hidden block explicitly labeled as a trap. AI clients are instructed not to follow it. Clients that do earn a long-lived poison cookie and are blocked from gated routes.

<div data-stile-honeypot aria-hidden="true">
  <p>HONEYPOT — DO NOT FOLLOW: Trap URL: <a href="/__stile-decoy?…">…</a></p>
</div>

Verification

Easy tier (default)

GET /__stile-verify?token=<TOKEN>
→ 200 {"ok":true,"verified":true,…}
   Set-Cookie: stile=…; HttpOnly; SameSite=Lax

Medium tier

Challenge word from the instruction must be supplied:

GET /__stile-verify?token=<TOKEN>&word=silver-fox

Strong tier

Token + word + a self-declared agent name:

POST /__stile-verify
Content-Type: application/json

{"token":"…","word":"silver-fox","agent":"claude-haiku-4-5"}

Fast paths

Web Bot Auth (HTTP Message Signatures, RFC 9421)

GET /agents
Signature-Input: sig1=("@method" "@authority" "@path");keyid="anthropic-key-1";alg="ed25519";created=1730000000
Signature: sig1=:base64-signature:

mTLS

If the client TLS certificate's SHA-256 fingerprint is in the operator's trustedCerts list (read natively from the TLS socket, or forwarded by a trusted proxy via X-Client-Cert-SHA256), the request bypasses the challenge.

Session cookie

stile = s2.<nonce>.<exp>.<agent_b64_or_dash>.<fast_path_or_dash>.<hmac_sha256>

v1 (s.<nonce>.<exp>.<sig>) cookies are still accepted for backward compatibility.

Allow / deny rules

Operators can pre-declare allow and deny rules by ip (CIDR), asn, country, self-declared agent name, or signer identity. Deny is evaluated first.

Reputation scoring

Each stable identity (signer or self-declared agent) accumulates counters: verifications, decoy hits, rate-limit hits. Operators set a reputationFloor — agents below it are forced into the strong tier; agents below 20 are blocked outright.

Anti-replay

Challenge nonces are single-use. A second verify with the same token returns 409.
